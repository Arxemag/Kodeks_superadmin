# Логика модуля init_company

Полное описание: что за чем выполняется, откуда берутся данные, где возможны расхождения и риски.

---

## 1. Входные данные (InitCompanyDTO)

Сообщение из Kafka или файл для пробы:

```json
{
  "id": 67,
  "reg": "350832",
  "companyName": "ООО Ромашка",
  "departments": [
    {
      "id": "a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11",
      "title": "TEST",
      "modules": ["Заведующий лабораторией", "Кабинет СМК"]
    }
  ]
}
```

- **id** — идентификатор компании (клиента).
- **reg** — номер регистрации, по нему ищем `base_url` в БД и логинимся в каталог.
- **companyName** — название компании (пишется в БД маппинга).
- **departments** — список отделов:
  - **id** — UUID отдела (используется как ключ в БД и при сборке ACL).
  - **title** — название **группы** в каталоге (например "TEST").
  - **modules** — названия **кабинетов** (инфобордов), например "Заведующий лабораторией", "Кабинет СМК".

---

## 2. Общий поток handle()

```
1. base_url по reg из БД (reg_services)
2. Синхронизация таблицы маппинга (department_service_mapping)
3. [dry_run → выход]
4. Логин в каталог (admin), получение cookies
5. Синхронизация групп в каталоге (создание недостающих)
6. Получение списка кабинетов (GraphQL)
7. Построение ACL-матрицы (кто к какому кабинету имеет доступ)
8. Применение ACL в каталоге (POST /admin/dirs)
```

По шагам ниже.

---

## 3. Шаг 1: base_url

- **Где:** `_get_base_url(reg)`.
- **Что:** `SELECT base_url FROM reg_services WHERE reg_number = :reg`.
- **Зачем:** Каталог у каждого reg свой (свой URL). Дальше все запросы к каталогу идут на этот `base_url`.

---

## 4. Шаг 2: Синхронизация таблицы маппинга (_sync_mapping_table)

- **Таблица:** `department_service_mapping` (или имя из `DB_TABLE_DEPARTMENT_MAPPING`).
- **Колонки:** id, department_id (UUID), service_group_name, reg, client_id, client_name, mapping_status.

**Логика:**

1. **3.1** Читаем все записи по `reg`: кто уже есть в БД.
2. **3.2** Для каждого отдела из payload:
   - если запись с таким `department_id` есть — делаем UPDATE (service_group_name, client_name, mapping_status = 'active');
   - если нет — INSERT (department_id, service_group_name = title, reg, client_id, client_name, mapping_status = 'active').
3. **3.3** Все отделы по этому reg, которых **нет** во входящем payload, помечаем как `mapping_status = 'archived'`.

**Итог:** В БД лежит актуальный список отделов компании: для каждого отдела — его UUID, **название группы** (title → service_group_name), reg и т.д. Список кабинетов (modules) в эту таблицу **не пишется** — он используется только при построении ACL (шаг 7).

---

## 5. Шаг 4: Синхронизация групп (_sync_groups)

- **Где:** GET страницы групп каталога, парсинг HTML, при необходимости создание групп.
- **Вход:** base_url, cookies, reg.
- **Выход:** `groups_dict` — словарь **название группы → id группы** (например `{"TEST": 342, "СМК": 530}`).

**Логика:**

1. GET `{base_url}/users/groups` → HTML.
2. Парсинг (parse_groups_catalog) → текущий список групп и их id.
3. Из БД берём `service_group_name` по reg для активных отделов (это как раз title из payload).
4. Для каждого такого имени, которого **нет** в каталоге, вызываем создание группы в каталоге (POST), затем снова GET groups и обновляем словарь.

**Зачем:** Чтобы у каждого отдела (по его title) в каталоге была группа с известным id. Эти id потом подставляются в ACL кабинетов.

---

## 6. Шаг 5.1: Получение кабинетов (_get_boards)

- **Где:** `get_boards()` в cabinet_acl — запрос к GraphQL каталога.
- **Запрос:** `board { many { id, title } }`, context=docs.
- **Выход:** `boards_dict` — словарь **название кабинета (title) → id кабинета** (например `{"Заведующий лабораторией": "P000A", "Кабинет СМК": "P000B"}`).

**Важно:** Ключи — это **человекочитаемые названия** из каталога (title). Id кабинета в разных местах могут быть в разном регистре (P000A / p000a); при сравнении мы приводим к нижнему.

---

## 7. Шаг 5.2: Построение ACL-матрицы (_build_acl_async)

- **Вход:**
  - reg;
  - `groups_dict` (название группы → id группы);
  - `boards_dict` (название кабинета → id кабинета);
  - `dep_id_to_modules` — из payload: для каждого department_id список названий кабинетов (modules).

- **Выход:** `acl_matrix` — словарь **id кабинета → список id групп**, которым даём доступ к этому кабинету.

**Логика:**

1. Из БД по reg читаем активные отделы: `(department_id, service_group_name)`.
2. Для каждой строки:
   - по `service_group_name` берём `group_id` из `groups_dict`;
   - по `department_id` из payload берём список `modules` (названия кабинетов);
   - для каждого названия кабинета из modules находим `board_id` в `boards_dict`;
   - добавляем в acl_matrix: для этого `board_id` в список групп добавляем `group_id`.
3. Убираем дубликаты групп по кабинету (порядок сохраняем).

**Пример:** Отдел "TEST" (group_id=342), modules=["Заведующий лабораторией"]. В каталоге "Заведующий лабораторией" → P000A. В итоге acl_matrix["P000A"] = [342]. То есть: кабинет с id P000A должен открываться группе 342.

**Возможные проблемы:**

- Если в каталоге кабинет называется иначе, чем в payload (modules), то `boards_dict.get(mod_title)` даёт None и этот кабинет в acl_matrix не попадёт.
- Ключи в acl_matrix — это id из GraphQL (например P000A). Дальше мы сверяем их с полями формы каталога (см. шаг 8).

---

## 8. Шаг 6: Применение ACL (_apply_acl)

Здесь мы должны отправить в каталог форму настроек DOCS так, чтобы в ней для нужных кабинетов стояли нужные группы доступа. Ожидается, что каталог принимает POST на `/admin/dirs` с теми же полями, которые сам отдаёт на странице настроек.

### 8.1 Получение формы (fetch_acl_form_via_http и fallback apply_acl_via_browser)

**Последовательность запросов (HTTP, как в add_cabinet_group):**

1. **GET** `{base_url}/admin/dirs`  
   Ответ — HTML страницы со ссылками на разделы (docs, users и т.д.). Из HTML парсим параметр **n** для перехода к docs (например n=2).

2. **GET** `{base_url}/admin/dir?n={n}`  
   Ответ — HTML формы настроек раздела (ещё без полей ACL кабинетов).

3. Собираем из этой формы все поля, подставляем дефолты из **SETUP_DIR_DEFAULTS** (path, type, to, com, grps_*, action и т.д.), выставляем **setup2 = "Настройки сервисов"**.

4. **POST** `{base_url}/admin/dir` с этой формой (application/x-www-form-urlencoded).  
   В ответе каталог отдаёт страницу с формой ACL. Если форма рендерится через JS, в HTML может не быть полей `acl_infoboard_*`.

5. **Если в ответе нет полей `acl_infoboard_*`** (форма только в браузере после JS): вызывается **apply_acl_via_browser** (Playwright). Открывается страница `/admin/dir?n={n}`, нажимается «Настройки сервисов», подставляются наши ACL в поля формы, форма отправляется из браузера («Сохранить»). Так обходится 500 при POST через httpx из-за проверки сессии каталогом. Требуется `pip install playwright` и `playwright install chromium`.

**Результат (HTTP-путь):** `setup_html` (HTML с формой ACL), `post_url`, `acl_form_base`, `docs_n`. При использовании браузера ACL уже применён, дальнейший POST через httpx не выполняется.

**Зачем так:** Получить актуальное состояние формы каталога и изменить только ACL кабинетов; при JS-форме — применить ACL в той же браузерной сессии.

### 8.2 Парсинг формы и список кабинетов

- **full_form** = `parse_full_form_from_html(setup_html)` — все input из HTML в виде словаря имя → value.
- Если в full_form нет поля **base**, подставляем его из `acl_form_base` (раскодируя %2Fdocs в /docs при необходимости).
- Если full_form пустой, пробуем собрать его из `parse_acl_from_html(setup_html)` (отдельные regex под acl_infoboard_*).

**Список кабинетов формы (form_board_keys):**

- Собираем ключи кабинетов из full_form: все имена, начинающиеся с `acl_infoboard_`, обрезаем префикс и суффикс `_view`, приводим к нижнему регистру → множество `form_board_keys`.
- Если это множество пустое (часто бывает, если общий парсер формы не видит acl-поля), делаем fallback: `parse_acl_from_html(setup_html)` и берём ключи оттуда; эти же поля дописываем в full_form как `acl_infoboard_{bk}` и `acl_infoboard_{bk}_view`, чтобы форма была полной.

**Зачем:** Каталог может отдавать форму, где acl-поля заданы в HTML так, что общий парсер их не видит, но parse_acl_from_html находит. Тогда мы не теряем кабинеты и не отправляем форму без acl-полей.

### 8.3 Безопасные переопределения ACL (acl_overrides_safe)

- В форму мы подставляем **только те** кабинеты, которые уже есть на странице каталога. Кабинеты из acl_matrix, которых нет в `form_board_keys`, в форму не добавляем (чтобы не слать лишние поля и не провоцировать 500 или порчу DOCS).
- **acl_overrides_safe** = из acl_matrix оставляем только записи, у которых ключ (id кабинета в нижнем регистре) входит в `form_board_keys`.
- Если в acl_matrix что-то было, но после фильтра ничего не осталось — пишем в лог предупреждение (какие кабинеты искали, какие доступны в форме).

**Зачем:** Ключи в acl_matrix — из GraphQL (например P000A). Ключи в форме каталога могут быть в другом регистре (p000a). Сравнение идёт по нижнему регистру. Мы не добавляем в форму кабинеты, которых каталог сам не отдал (не передаём extra_board_ids и не подставляем "новые" acl_infoboard_*).

### 8.4 Сборка итоговой формы (build_admin_dirs_form)

- **Вход:** full_form (все поля с страницы + при необходимости дополненные из parse_acl_from_html), acl_overrides_safe (только кабинеты из формы), **extra_board_ids = None** (дополнительные кабинеты из GraphQL не добавляем).

**В build_admin_dirs_form:**

1. Копируем full_form, подставляем недостающие системные поля из _BASE_PARAMS_FALLBACK, выставляем **cmd = "save"**.
2. Собираем множество кабинетов **all_keys**: все ключи из формы (по полям acl_infoboard_*) + ключи из acl_overrides. extra_board_ids не используем.
3. Для каждого кабинета из all_keys:
   - если он есть в acl_overrides_safe — значение ACL берём оттуда (список group_id в JSON вида `{"kw":["342","530"]}`);
   - иначе — оставляем значение из формы или пустой `{"kw":[]}`;
   - для кабинетов с непустым ACL выставляем чекбоксы `infoboard_{bk}` и `infoboard_{bk}_view` = "on".
4. Возвращаем форму как список пар (имя, значение).

**Важно:** Пустые поля из формы **не выкидываем** — их отсутствие в запросе каталог может трактовать как сброс настроек и портить DOCS.

### 8.5 Отправка POST /admin/dirs (только при HTTP-пути)

- Если использовался **apply_acl_via_browser**, форма уже отправлена из браузера, этот шаг не выполняется.
- **URL:** post_url (из fetch_acl_form_via_http).
- **Тело:** urlencode списка пар (то же, что отдал build_admin_dirs_form), Content-Type: application/x-www-form-urlencoded.
- **Заголовки:** Referer = `{base_url}/admin/dir`, Origin = base_url (как в add_cabinet_group).
- **Повторы:** при 5xx делаем несколько попыток с экспоненциальной задержкой (USERS_RETRY_*). При последней неудаче пишем в лог и при наличии DUMP_ADMIN_DIRS_ON_5XX сохраняем тело запроса в файл.

## 9. Где могут быть расхождения и что проверить

1. **Названия кабинетов (modules) vs каталог**  
   В payload в modules приходят строки (например "Заведующий лабораторией"). Они ищутся в `boards_dict` по ключу **title** из GraphQL. Если в каталоге title кабинета написан иначе (пробелы, регистр, сокращения), маппинг не найдётся и этот кабинет не попадёт в acl_matrix.

2. **Id кабинета: GraphQL vs форма**  
   GraphQL отдаёт id в одном регистре (например P000A), форма в HTML может использовать другой (p000a). Мы сравниваем через .lower(), но если в форме кабинеты названы по-другому (не id, а иначе), то form_board_keys и acl_matrix не совпадут и будет предупреждение "acl_matrix кабинеты ['P000A'] не найдены в форме каталога".

3. **Форма без acl_infoboard_* в HTML (JS-рендеринг)**  
   Если страница настроек рендерит acl-поля через JS, в ответе HTTP их нет. В этом случае вызывается **apply_acl_via_browser** (Playwright): форма открывается в браузере, ACL подставляются и отправляются из той же сессии. Без установленного Playwright в логах будет предупреждение и попытка отправить минимальную форму через httpx (может привести к 500).

4. **Полнота формы и 500**  
   Каталог может требовать строго тот набор полей, который сам отдал. Мы не добавляем лишние кабинеты (extra_board_ids=None), не выкидываем пустые поля. Если 500 сохраняется, имеет смысл сравнить сохранённое тело запроса (admin_dirs_5xx_request.txt) с тем, что реально отправляет браузер при ручном сохранении настроек (те же имена полей, порядок, наличие path/action/base и т.д.).

5. **Один и тот же flow, что и add_cabinet_group**  
   Получение формы делается через fetch_acl_form_via_http (GET dirs → GET dir?n → POST dir с setup2 и SETUP_DIR_DEFAULTS). Если add_cabinet_group у вас отрабатывает, а init_company — нет, различие только в том, какие кабинеты и группы мы подставляем (несколько кабинетов/групп вместо одного) и в том, что мы не передаём extra_board_ids. Сравнение сохранённого запроса init_company с запросом add_cabinet_group (например с --dump-form) помогает увидеть отличия.

## 10. Краткая схема данных

```
Payload (departments[].title, .modules)
       ↓
БД department_service_mapping: department_id, service_group_name (= title)
       ↓
Группы каталога: title → group_id  (GET /users/groups)
Кабинеты каталога: title → board_id  (GraphQL board.many)
       ↓
acl_matrix: board_id → [group_id]  (по отделам из БД и их modules)
       ↓
Форма каталога: acl_infoboard_{board_id}, acl_infoboard_{board_id}_view
       ↓
Подставляем только кабинеты из формы; POST /admin/dirs с полной формой (без лишних полей, без удаления пустых).
```

Если нужно, можно добавить в этот документ примеры конкретных значений (reg, payload, фрагменты full_form и acl_matrix) для отладки под вашу среду.
